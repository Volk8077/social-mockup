Workflow: จากคอมเมนต์สู่การตอบกลับ

ทุกขั้นตอนที่เกิดขึ้นเมื่อมีคอมเมนต์ใหม่บนเพจของลูกค้า — สำหรับใช้ภายในทีม

เครื่องมือ

ลูกค้า / Facebook user Meta Graph API Make.com Airtable Claude API Softr Portal LINE Messaging API

ระดับความเสี่ยง

● ต่ำ — ตอบอัตโนมัติ ● กลาง — รอ B อนุมัติ ● สูง — ส่ง LINE ให้ B จัดการเอง
ส่วนที่ 1 — เส้นทางร่วม (ทุกคอมเมนต์ผ่านขั้นตอนนี้)
1
ลูกค้าคอมเมนต์
ลูกค้า
ลูกค้าใหม่หรือเก่าพิมพ์คอมเมนต์ใต้โพสต์ของ Company B บน Facebook หรือ Instagram
2
Meta ส่ง webhook ไปที่ Make.com
Meta API
เพจของ B subscribe กับ App ของ Company A ไว้แล้ว Meta จึงยิง POST ไปที่ webhook URL ของ Make.com ทันทีที่มีคอมเมนต์ใหม่
payload: { comment_id, comment_text, post_id, page_id, sender_name, sender_id, created_time }
3
Make.com หาว่าเป็นลูกค้ารายไหน
Make.com
ใช้ page_id ที่ได้จาก webhook ค้นหาใน Airtable [Clients] เพื่อดึง config ของลูกค้ารายนั้น
→ Airtable [Clients] · find by page_id · returns: client_id, brand_voice, escalation_rules, line_user_id
4
ดึง Knowledge Base ของลูกค้ารายนี้
Airtable
ดึง KB items ที่เกี่ยวข้อง — สำหรับ v1 ดึงทุก item ของ client_id นั้นไปใส่ใน prompt; v2 ใช้ semantic search (RAG)
→ Airtable [KB_Items] · filter client_id = X · returns: array of FAQ items
5
Make.com เรียก Claude API
Claude
รวม brand voice + KB + comment text เข้าด้วยกันแล้วส่ง POST ไปที่ api.anthropic.com/v1/messages ขอ structured JSON output
prompt: "You are replying as <brand voice>. KB: [...]. Customer comment: <text>. Return JSON: {risk, draft, reasoning}"
6
Claude ตอบกลับ — ระดับความเสี่ยง + ฉบับร่าง
Claude
Claude คืนค่าเป็น JSON บอก risk tier (low/medium/high), ฉบับร่างคำตอบ (เฉพาะ low/medium), และเหตุผลเสมอ — Make ใช้ field risk เพื่อแยกเส้นทางต่อไป
response: { "risk": "medium", "draft": "...", "reasoning": "Customer asking about size availability — factual KB question" }
ส่วนที่ 2 — แยกตามระดับความเสี่ยง
● ความเสี่ยงต่ำ
เช่น "สวยมาก!", "เปิดกี่โมง?" — Claude ตอบให้ ส่งทันที ไม่ต้องรออนุมัติ
1
โพสต์คำตอบกลับ FB Meta API
Make เรียก POST /{comment-id}/comments ใช้ access token ของเพจ B
2
บันทึกลง Airtable Airtable
เพิ่ม row ใน [Comments_Queue] · status = sent · tier = low
3
โผล่ในแท็บ "ตอบอัตโนมัติแล้ว" Softr
Softr ดึงข้อมูลจาก Airtable แบบ real-time · B เห็นได้ทันที สำหรับ audit
● ความเสี่ยงกลาง
เช่น "มีสีอื่นมั้ย?", "ส่งต่างประเทศมั้ย?" — Claude ร่างคำตอบไว้ B กดอนุมัติหรือแก้ไขก่อนส่ง
1
บันทึกฉบับร่างลง Airtable Airtable
[Comments_Queue] · status = pending_approval · มี field draft_text ที่แก้ไขได้
2
B เห็นในแท็บ "รออนุมัติ" Softr
Softr เปิด form ให้ B กดอนุมัติ / แก้ไข / ปฏิเสธ ได้ทันที
3
B กด "อนุมัติและส่ง" Softr
Softr อัปเดต Airtable: status = approved · เก็บข้อความที่ B แก้ไขแล้วใน final_text
4
Make จับการเปลี่ยนแปลง → ส่งคำตอบ Make Meta API
Make scenario อีกอันที่ watch ตาราง [Comments_Queue] → เจอ status=approved → โพสต์ผ่าน Meta API → อัปเดต status = sent
● ความเสี่ยงสูง
เช่น คำร้องเรียน, ขอเงินคืน, กล่าวหาของปลอม, ภาษากฎหมาย — AI ไม่ตอบ ส่งให้ B จัดการเอง
1
บันทึกใน Airtable + เหตุผล Airtable
[Comments_Queue] · status = needs_attention · ไม่มี draft · มี ai_reasoning อธิบายว่าทำไมต้องการ B
2
ส่งแจ้งเตือน LINE LINE API
Make เรียก LINE Messaging API → push ข้อความไปที่ line_user_id ของ B พร้อมลิงก์เปิด portal
3
B เห็นทั้งใน LINE และพอร์ทัล Softr
แท็บ "ต้องดูด่วน" แสดงคอมเมนต์ + เหตุผล + ปุ่ม "ตอบบน Facebook"
4
B จัดการเองบน FB
B ตอบคอมเมนต์โดยตรงบน Facebook ในเสียงของแบรนด์ตัวเอง · ระบบไม่โพสต์ให้
5
B กด "ทำเครื่องหมายว่าจัดการแล้ว" Softr
Softr อัปเดต Airtable: status = handled · นำออกจากแท็บ "ต้องดูด่วน"
ส่วนที่ 3 — รวมข้อมูลทั้งหมด
📊

คอมเมนต์ทั้งหมด เก็บใน Airtable [Comments_Queue]

ไม่ว่าจะระดับไหน ทุก row ถูกเก็บถาวรพร้อม timestamp · risk tier · final_text · เพื่อใช้ทำ analytics ภายในของ Company A และเป็น audit log ให้ลูกค้าดูได้

ภาคผนวก — โครงสร้างตาราง Airtable

Airtable schema ที่ใช้ในเส้นทางนี้

4 ตารางหลัก ออกแบบให้ multi-tenant — เพิ่มลูกค้าใหม่คือเพิ่ม row ใน [Clients] และ KB ของพวกเขา ที่เหลือใช้ pipeline เดิมได้เลย

Clients
ข้อมูล config ของลูกค้าแต่ละราย (Company B, C, D…)
client_id · name · page_id (FB) · ig_account_id · brand_voice · escalation_rules · line_user_id · subscription_tier · setup_date
KB_Items
FAQ และข้อมูลแบรนด์ของลูกค้าแต่ละราย
kb_id · client_id (link) · category · question · answer · last_updated · is_active
Comments_Queue
คอมเมนต์ทุกอันที่เข้าระบบ — heart ของ workflow นี้
comment_id · client_id (link) · fb_comment_id · sender_name · comment_text · risk_tier · ai_draft · final_text · ai_reasoning · status · created_at · sent_at
Posts
ตารางโพสต์ที่จะอัปโหลด (ใช้ใน Build 2)
post_id · client_id (link) · scheduled_time · content · image_url · platform · status · posted_at · post_url
